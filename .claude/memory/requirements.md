## CAT — Catalog & Track Data

### Track Catalog

- CAT_001: Display two sections in the catalog panel: "My Tracks" (local) and "Catalog Tracks" (remote/pre-built).
- CAT_002: Show a loading state while the remote catalog index is being fetched.
- CAT_003: Show an error message in the catalog panel if the remote catalog fetch fails.
- CAT_004: Each catalog entry displays the track name and its ID.
- CAT_005: Allow selecting a track by clicking its entry to load it into the workspace.
- CAT_006: Remote tracks have no delete action; only local tracks show a delete button.
- CAT_007: Allow creating a new local track via a "+" button in the "My Tracks" section.
- CAT_008: New local tracks are seeded from the catalog track `elements/ladder3` — creating a new track is equivalent to cloning `elements/ladder3` with an auto-generated name.
- CAT_009: New local tracks are auto-named with an incrementing suffix (e.g., `track-001`, `track-002`).
- CAT_010: After creation, the app immediately navigates to the new track.
- CAT_011: Allow deleting a local track via a delete button on its entry, with a confirmation step.
- CAT_012: If the deleted track is currently open, navigate to home after deletion; otherwise remain on the current track.
- CAT_013: Every catalog entry (local and remote) displays a clone button.
- CAT_014: Cloning a track creates a new local track with the source track's data and an auto-generated name following the same incrementing scheme as CAT_009.
- CAT_015: After cloning, the app immediately navigates to the cloned track.

### Track Data

- CAT_016: A track is defined by `edges` (structural segments) and `path` (racing route through gates).
- CAT_017: Each point is a `Point3 = [x, y, z]` triple of integers on the 3D lattice grid.
- CAT_018: An edge is a `[Point3, Point3]` pair representing a structural line segment.
- CAT_019: A path is a sequence of gate-transition steps (`PathStep[]`), each step being `{ gates: TrackSegment[]; entry?: Direction }` (see VIZ_021).
- CAT_020: Tracks may include optional metadata: `name`, `description`.
- CAT_021: Tracks may include an optional `caption` overlay: `{ enabled, text_1, text_2, location, orientation_angle? }`.
- CAT_022: Tracks may include a `show_path_labels` boolean to toggle path segment label display in the 3D view.
- CAT_023: Remote tracks are served as static JSON files listed in a plain-text index file (`/tracks/tracks.txt`).
- CAT_024: Local tracks are persisted in `localStorage` under versioned key `fpv-track-builder.local-tracks.v1`.
- CAT_025: On reading local tracks from `localStorage`, validate the data structure and fall back to an empty list on parse or schema errors.
- CAT_026: Remote tracks are fetched lazily — individual track JSON is loaded only when selected, not at startup.
- CAT_027: Loaded remote tracks are cached in-memory; duplicate concurrent requests for the same track are deduplicated.
- CAT_028: `public/tracks/tracks.txt` is auto-regenerated by a pre-commit git hook (`.git/hooks/pre-commit`): it finds all `*.json` files under `public/tracks/`, sorts them by path, writes one relative path per line, and stages the result. Must never be edited by hand.
- CAT_029: Remote catalog tracks are organized into subdirectories under `public/tracks/` (e.g. `public/tracks/RG5/rg5-06.json`). A track's ID is its path relative to `public/tracks/` without the `.json` extension (e.g. `RG5/rg5-06`).
- CAT_030: The catalog panel renders remote tracks as a tree mirroring the directory hierarchy — each subdirectory appears as a collapsible group containing its tracks.

### Stores

- CAT_031: `src/store/tracks.store.ts` is a single unified Zustand store holding all track data with no distinction between local and remote origin.
- CAT_032: `src/store/metadata.store.ts` is a Zustand store holding per-track metadata: `readonly` flag, remote load status (`idle | loading | loaded | error`), and load error message.
- CAT_033: The `readonly` flag in the metadata store identifies remote tracks; local tracks have `readonly: false`.
- CAT_034: Remote track load status and errors are managed exclusively in the metadata store, not the tracks store.

## WS — Workspace

### Layout

- WS_001: The workspace uses a three-panel resizable layout: catalog (left, collapsible), editor (center), debug (right, dev-only).
- WS_002: The left catalog panel can be collapsed and expanded via a toggle button.
- WS_003: Panel collapse state is session-only and resets on page reload.

### Corner Buttons

- WS_015: A `CornerButtons` component renders three fixed-position icon buttons anchored to the bottom-left of the viewport, always on top of other content.
- WS_016: The three buttons in left-to-right order are: MeflyNav dots menu, GitHub repository link, help button.
- WS_017: MeflyNav dots menu uses `<MeflyNavReceiver>` from the `mefly-nav` package with `activationMode="hover"`, positioned at `left: 1rem, bottom: 1rem`. Its appearance is controlled via `--mefly-nav-*` CSS variables mapped from `--tb-*` tokens.
- WS_018: The GitHub repository link opens the project's GitHub repository in a new tab and renders the GitHub SVG icon (inline).
- WS_019: The help button renders a question-mark SVG icon and pulses with a glow animation until the user clicks it for the first time. The "seen" state is persisted in `localStorage` under key `fpv-track-builder.v1.helpSeen`.
- WS_020: When the app is embedded in an iframe (`window.parent !== window`), the MeflyNav occupies the leftmost slot and the repo link and help button shift right by one slot width to avoid overlap.

### TrackEditor Component

- WS_011: The center panel hosts a `TrackEditor` component (`src/features/trackEditor/TrackEditor.tsx`).
- WS_012: `TrackEditor` is composed of a toolbar (`TrackEditorToolbar.tsx`) and a scene panel (`TrackEditorScene.tsx`), both co-located in `src/features/trackEditor/`.
- WS_013: `TrackEditor` dispatches toolbar events down to `TrackEditorScene`; `TrackEditorScene` owns all mode-specific rendering logic.
- WS_014: `TrackEditorScene` is initialized with a ref to the current track entry in the tracks store and re-renders when that entry changes.

### SidePanel Component

- WS_004: Both the catalog panel and the debug panel are implemented using a shared `SidePanel` component (`src/layout/SidePanel.tsx`).
- WS_005: `SidePanel` accepts a `position` prop with values `'left'` or `'right'`.
- WS_006: `SidePanel` accepts a `minSize` prop (number) defining the minimum width before the panel auto-collapses.
- WS_007: `SidePanel` accepts a `defaultSize` prop and a `collapsedSize` prop, both forwarded to the underlying `react-resizable-panels` panel.
- WS_008: `SidePanel` accepts a `caption` prop (string) rendered as a heading at the top of the panel when expanded.
- WS_009: When the panel is collapsed, the caption is rendered vertically instead of horizontally.
- WS_010: For a left-positioned panel the collapsed caption reads upward (bottom-to-top); for a right-positioned panel it reads downward (top-to-bottom).

## VIZ — Visualization & Editors

### 3D Visualization

- VIZ_001: Render track edges as solid pipes on a 3D lattice grid using Three.js.
- VIZ_002: Render the racing path as a distinct highlighted route through gates (translucent gate tiles).
- VIZ_003: Display a step number for each path step, anchored inside the top-right corner of the step's first gate. Numbers are always shown in view and path modes; the `show_path_labels` track flag is retained in the data model but no longer gates display.
- VIZ_004: Support interactive camera controls: orbit, zoom, and pan.
- VIZ_005: The 3D canvas uses a dark background to contrast with track elements.

### Flight Animation

- VIZ_019: In view mode (Track Viewer), animate a quadcopter flying the racing path. The flight curve threads the *first* gate of each path step (one gate per step), in step order, as a closed Catmull-Rom curve looped continuously. A step with an `entry` direction (VIZ_021) contributes a *pierce pair* of control points straddling the gate center along the entry direction, forcing the curve to cross that gate perpendicular and on the intended side instead of cutting the shortest diagonal; an undirected step contributes just the gate center. The gate-center waypoints may additionally be corner-rounded by a tunable fillet radius (clamped per-edge to half its length) to soften turns. Requires at least two steps with a gate; otherwise no drone is shown. The quad is a small stylized model oriented along its direction of travel.
- VIZ_020: Render the flight path as a thin line at a low resting opacity, overlaid with a trail that brightens behind the moving drone — maximum opacity at the drone, fading back to the resting opacity over the distance the drone covers in the trailing 4 seconds.

### Workspace Modes

- VIZ_006: The center panel displays one of four modes selected via the `?mode=` URL query parameter: `view` (Track Viewer), `gates` (Gates Editor), `path` (Path Editor), `json` (JSON View).
- VIZ_007: Default mode is `view` when the `mode` parameter is absent from the URL.
- VIZ_008: Clicking a mode button in the toolbar updates the `?mode=` query parameter without reloading the page.
- VIZ_009: In `json` mode the center panel renders the selected track's data as a read-only JSON object tree instead of the 3D scene.

### Gates Editor

- VIZ_010: Render placed edges as solid cylinder "pipes". Alongside them, render candidate edges — every unit axis-aligned lattice edge that starts or ends at a node touched by a placed edge — as half-opacity thin lines. Candidates never extend below the floor (no endpoint with y < 0). An empty track seeds candidates around the origin.
- VIZ_011: Clicking a candidate edge places it (adds it to the track's edges).
- VIZ_012: Clicking a placed edge removes it (it becomes a candidate again).
- VIZ_013: Persist all gate edits back to the track store immediately.
- VIZ_018: Hovering any placed or candidate edge highlights it (warning-color tint, pointer cursor) to show what a click will toggle.

### Path Editor

- VIZ_014: A path step contains one or more gates. A gate is a 1×1 axis-aligned lattice plane, stored as a diagonal segment (endpoints differ by ±1 on exactly two axes). All gates within a step lie in the same plane and each is edge-connected to another gate of the step. Gates render as translucent filled quads with outlines.
- VIZ_015: Allow reordering path steps via the steps panel.
- VIZ_016: Step creation: hovering a placed edge highlights it; clicking selects it (selecting another deselects). The four gates sharing the selected edge appear as candidates; hovering a candidate highlights it. Left-click adds the gate and finishes the step; right-click adds it and offers the in-plane neighbors of the last gate as further candidates (excluding gates already in the draft, gates whose shared side with the last gate is a placed edge — a pipe divides the opening — and unanchored gates — a candidate must touch the structure with at least one corner on a node of a placed edge, which keeps every gate at most one tile from the track while allowing steps to wrap around the structure in shoulder shapes). Stepping straight back across the initiating edge is blocked because that edge is placed, but a step that wraps around the structure may still reach the gate on its far side from another direction. Gates lying flat on the floor (y = 0 plane) or with any corner below it are never offered. Click-away finishes an open draft; Escape discards it. Steps are removed via the panel ✕ or the Delete key.
- VIZ_017: Persist all path edits back to the track store immediately.
- VIZ_021: A path step is an object `{ gates: TrackSegment[]; entry?: Direction }` (the legacy bare `TrackSegment[]` step shape is no longer supported — catalog JSON and the data model use the object form). `entry` is a signed lattice axis (`up`/`down`/`left`/`right`/`forward`/`backward`, y-up; `z+` is `backward`) constrained to the step's gate normal axis, recording which side is the entrance. It is chosen when the step's *first* gate is placed: while the draft is empty, hovering a candidate gate draws an arrow into it from the camera-facing side (you enter from the side you look from), updated per frame so it tracks orbiting. Left-click commits the gate and its direction; right-click settles the direction on the first gate and then stops drawing the hover arrow as further in-plane gates are added. Placed steps that carry an entry render a static direction arrow. The flight path honors the entry direction (VIZ_019).

## ETC — Debug, Embedding & Export

### Debug Tools

- ETC_001: Show a debug panel (right side) only in development mode (`import.meta.env.DEV`).
- ETC_002: The debug panel provides interactive JSON editors for: tracks store state, metadata store state, the currently loaded track, and layout store state.
- ETC_003: Editing the current track's JSON in the debug panel updates the live track state immediately.
- ETC_004: The debug panel displays a "Debug Tools" heading and the currently selected track ID (or "none" when no track is selected).
- ETC_005: The panel contains four `<JsonEditor>` sections (from `json-edit-react`) stacked vertically. Each editor's object tree is collapsed by default — the root node is visible but all children start collapsed.
- ETC_006: The tracks store editor (rootName `tracksStore`) is read-only; it exposes the full store state.
- ETC_007: The metadata store editor (rootName `metadataStore`) is read-only; in-flight load promises are serialized as a boolean flag plus an array of currently in-flight track IDs for readability.
- ETC_008: The current track editor (rootName `trackStore`) is editable; any change made through the editor is applied back to the tracks store immediately.
- ETC_009: The layout store editor (rootName `layoutStore`) is read-only; it is the last editor in the panel.

### URL & Embedding

- ETC_010: Track selection and workspace mode are encoded in the URL: `/#/tracks/{id}?mode=...`.
- ETC_011: The app supports iframe embedding; route changes are synced with the parent window via `postMessage`.
- ETC_012: The parent window can send `NAVIGATE_TO_HASH` messages to navigate the embedded app.
- ETC_013: The embedded app sends `HASH_CHANGED` messages to the parent window on every route change.

### Export & Sharing

- ETC_014: Allow exporting the current track as a JSON file.
- ETC_015: Allow copying a shareable URL or embed code for the current track.
